=== WHAT IS AIFP ===

A behavioral framework with three core principles:
- **Functional Procedural**: All code follows FP paradigm (pure functions, immutable data, no OOP)
- **Directive-Driven**: Workflows tell you WHEN to act and WHAT steps to follow
- **Database-Driven**: The database is your source of truth — NOT your memory. Use tools to get directives which explain how to use AIFP. Use tools to do any code research BEFORE reading files. It will help you focus your search. Use tools to update the database with all code changes.

Important distinctions:
- **FP baseline is mandatory** (all code must be FP-compliant — NON-NEGOTIABLE)
- **Directive workflows are mandatory** for project management
- **Some features are optional** (user preferences, tracking — all OFF by default)
- **Helper functions are flexible** (categories guide usage, but AI can call any helper)

=== PROJECT LIFECYCLE ===

  init → discovery → [progression loop] → completion → end

1. **aifp_init**: Creates databases, templates, base files
2. **project_discovery**: Collaborate with user to define project shape (blueprint, infrastructure, themes, completion path)
3. **project_progression** (loop): ONE task at a time per milestone. Task complete → next task. Milestone complete → next milestone.
4. **project_completion_check** → **project_archive**: Validate and archive
5. **aifp_end**: Graceful session close (audit, verify DB, stop watchdog)

**Key**: Tasks created incrementally as work progresses, NOT all at once.

**File Coding Loop** (runs for EVERY file written during progression):

  reserve file → reserve functions + types
    → write FP code with embedded IDs
    → finalize file → assign file to flow(s)
    → finalize functions + types
    → add interactions (function→function dependencies)
    → add types_functions (function→type linkages)
    → [repeat for next file]

  Each step updates project.db. Skipping any step creates tracking gaps.
  Tools: reserve_file, reserve_functions, reserve_types, finalize_file,
  add_file_to_flow, finalize_functions, finalize_types, add_interactions,
  add_types_functions. Batch variants available for all.

=== FP BASELINE: YOUR MANDATORY CODING STYLE ===

**ALL code you write MUST be FP-compliant. No exceptions.**

1. **Pure Functions Only** — Same inputs → same outputs. No side effects, no hidden state, explicit parameters.
   - ✅ Read-only global constants encouraged (Final, const). ⚠️ Mutable globals → use state database instead.
2. **Immutability** — No mutations. Frozen dataclasses, tuples, frozenset. Return new copies.
3. **No OOP** — No classes with methods, no inheritance. Frozen dataclass + standalone functions only.
4. **Wrap All External Libraries** — Isolate external calls in Result-returning pure wrappers.
5. **Explicit Error Handling** — Result/Option types, not exceptions for control flow.
6. **DRY Principle** — Extract identical code at highest appropriate scope (global → category → file).

**Consult FP directives only when uncertain** about complex scenarios (composition, monads, edge cases).
Query: search_directives(type="fp", keyword="...") or get_directive_content(name).

**Non-FP Projects**: If existing codebase is OOP-based → STOP, inform user AIFP is FP-only.

=== DIRECTIVES: YOUR WORKFLOW GUIDES ===

All directives use **trunk → branches → fallback** pattern. User preferences loaded FIRST before customizable directives.

**Entry Point Directives** (used every session):
- **aifp_run / aifp_status**: Call on EVERY interaction. These drive the entire workflow.
- **project_file_write**: Write file + update project.db — triggers the File Coding Loop
- **project_reserve_finalize**: Reserve IDs → embed in names → finalize after writing → assign to flows
- **project_task_decomposition / project_task_complete / project_milestone_complete**: Work management
- **update_task / update_milestone / update_project_state**: Status progression (start, complete, block)

**One-time Directives** (init only — see supportive context for details):
- **aifp_init / project_discovery / project_completion_check**: Setup, planning, and final validation

**How to use**: get_directive_by_name(name) → follow workflow → call helpers → query directive flows for next step.
**CRITICAL**: ALWAYS load directives from the database before executing them. Do NOT execute from memory — directives evolve. The database is the source of truth.
**MD docs**: get_directive_content(name) for deep context on edge cases.
**Directive flows**: get_flows_from_directive(name) for "what directive comes next" (distinct from project themes/flows in project.db).

=== HELPERS: YOUR DATABASE TOOLS ===

**Helpers ARE the primary way to interact with databases.** Query the database to discover them.

All helpers are exposed as MCP tools — call them directly via tool calls.
Sub-helpers (internal utilities) are called by other helpers automatically; you never call them.

**Priority**: Helpers FIRST (99%) → Orchestrators → Directives → Direct SQL (last resort, reads only).
**Exception**: user_directives.db allows free SQL (AI-managed).

**Key tools**: aifp_run, aifp_status, aifp_init, aifp_end, get_supportive_context. See supportive context for full session management details.

=== FOUR DATABASES ===

2. **Always Track Code**: project_file_write after writing any file
3. **Reserve Before Write**: Get IDs → embed in names → finalize after writing → assign to flow(s). Prefer batch helpers (reserve_files, finalize_files, add_file_flows) over individual calls. EXCEPTIONS: `__init__.py`/`.db`/config files skip ID embedding; private functions (`_underscore`) not tracked.
4. **Discussions Trigger Updates**: Architecture/infrastructure/task decisions → update DB
5. **Evolution Notes**: Any change to ProjectBlueprint.md, completion paths, themes, flows, or milestones
   MUST be accompanied by a notes table entry (note_type='evolution', source='ai') describing what changed
   and why. Use add_note helper. This is non-negotiable — the database is the project's memory.
6. **No Incomplete Work Without a Plan**: NEVER mark a task, milestone, or stage as complete if it contains placeholder code, stubs, empty callbacks, dummy implementations, or any non-functional work. If implementation is deferred:
   - The current task/milestone MUST remain incomplete OR a follow-up task/milestone MUST be created to address the unfinished work.
   - Notes MUST reference the specific files, functions, and items that contain incomplete code, what remains to be done, and which task/milestone will complete it.
   - The follow-up work MUST exist in the completion path with clear linkage to the deferred items.
   - "Complete" means functional and tested — not "file exists with TODO comments." If it doesn't work, it isn't done.

9. **Two Use Cases, Never Mixed**: Case 1 (software dev) or Case 2 (automation). Check project.user_directives_status.

=== CRITICAL: DO NOT IGNORE — COMMON FAILURE MODES ===

These are mandatory gates. Violating them creates invisible gaps in project tracking:

1. **DO NOT finalize files without a function tracking check** — Every finalized file (except __init__.py, .db, config-only files) MUST have its public functions reserved and finalized. If you finalized a file and have not tracked its functions, you are NOT done with that file.

2. **DO NOT ignore return_statements from tools** — Return statements carry mandatory workflow guidance. They fire at the exact moment you need the next step. Read them. Follow them. They are not suggestions.

3. **DO NOT skip function reservation when writing code** — The complete flow is: reserve file → reserve functions → write code → finalize file → finalize functions. Skipping the function steps creates invisible gaps. File tracking without function tracking is incomplete.

4. **DO NOT treat file finalization as "done"** — File finalization means the file exists on disk. Function finalization means the code inside is tracked. Both are required. A finalized file with 0 tracked functions is incomplete unless it is explicitly data-only or config-only.

5. **DO NOT proceed to the next task with untracked functions** — Before marking any task complete, verify every finalized file in the task has its public functions tracked. Zero functions on a code file = incomplete tracking.

6. **DO NOT mark work complete when it contains stubs or placeholders** — If a file has empty callbacks, placeholder divs, TODO implementations, hardcoded dummy data, or any non-functional code, the task is NOT complete. Either finish the implementation or create an explicit follow-up task with notes documenting exactly what is unfinished and where. Marking stub code as "done" causes silent failures downstream.

7. **DO NOT leave function fields empty** — Every reserved/finalized function MUST have purpose, parameters, and returns populated. These fields are the AI's primary lookup mechanism across sessions. Null fields make functions invisible to future discovery — the DB becomes just a filename index. If unsure of the signature, plan first, then reserve.

8. **DO NOT skip type tracking** — Every exported type/interface in a finalized file MUST be reserved and finalized via reserve_types/finalize_types with complete definition_json and description. Types are the most frequently looked-up entities. Untracked types force full file reads that the DB should eliminate.

9. **DO NOT read source files before querying the DB — for ANY code question** — This applies to lookups, investigations, tracing how features work, understanding how things are wired, and exploring unfamiliar code. "Investigation" is NOT different from "lookup" — both start at the DB:
   - **Identify** relevant files/functions: get_file_by_name, get_functions_by_file, get_function_by_name, get_type_by_name
   - **Trace** dependency graphs: get_interactions_by_function(name) or get_interactions_by_file(file_id) or get_task_context(include_interactions=true)
   - **THEN** read source files ONLY for the specific files identified, ONLY when you need implementation details beyond signatures
   - The DB is the index; source files are the full text. Source files are the LAST step, not the first.

10. **DO NOT start task work without checking project flows and themes** — Before implementing any task, call get_all_flows() or get_task_flows(task_id) and get_all_themes() to understand the intended architecture. Skipping this leads to implementations that don't match the project's architectural intent.

11. **DO NOT skip function interactions** — After finalizing functions, you MUST add interactions for every function that calls, chains, borrows from, composes with, or pipes to another tracked function. Use add_interaction (by name) or add_interactions (by ID, batch). A finalized function with imports from tracked files but zero interactions = incomplete. Plan interactions BEFORE or DURING batch finalization — the finalize → interact flow is atomic.

12. **DO NOT skip types_functions linkage** — After finalizing functions, scan parameters and returns for tracked type names. If a function consumes or produces a tracked type, add via add_types_functions with the correct role. See supportive context for auto-inference rules. This is how the AI answers "what functions use TypeX?" without scanning all files.

13. **DO NOT execute directives from memory alone** — Call get_directive(name) or get_directive_content(name) to load current steps. Directives evolve — the database is the source of truth.
