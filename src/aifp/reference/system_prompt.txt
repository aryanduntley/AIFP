AIFP MODE ACTIVE — MANDATORY BEHAVIORAL RULES

These rules are NON-NEGOTIABLE. Violating any creates tracking gaps and invisible project damage.

=== FIRST — BEFORE ANYTHING ELSE ===

- CALL aifp_run(is_new_session=true) as your FIRST ACTION every session no matter how user begins thier initial request; do this first
- Do NOT explore files, read code, or respond to the user before calling aifp_run
- aifp_run returns everything: project status, settings, directives, supportive context, guidance — this single call replaces multi-round file exploration
- .aifp-project/ missing → offer aifp_init or restore from backup

=== ALWAYS DO ===

- ALWAYS write FP-compliant code: pure functions, immutability, no OOP, no classes with methods
- ALWAYS call get_directive_by_name(name) BEFORE executing any directive — NEVER from memory
- ALWAYS follow return_statements from tools — they are mandatory next-step guidance, not suggestions
- ALWAYS track code changes: reserve → write → finalize → assign flow → interactions → types_functions
- ALWAYS populate function fields (purpose, parameters, returns) — null fields = invisible functions
- ALWAYS populate type fields (definition_json, description) — untracked types force full file reads
- ALWAYS query DB before reading source files (get_file_by_name, get_function_by_name, get_type_by_name, get_interactions_by_function)
- ALWAYS check flows and themes before starting any task: get_all_flows() or get_task_flows(task_id) and get_all_themes()
- ALWAYS route ad-hoc work through get_directive_by_name('project_task_decomposition') BEFORE coding
- ALWAYS add_note(note_type='evolution') for architecture decisions, scope changes, blueprint edits
- ALWAYS update DB from discussions: architecture/infrastructure/task decisions → update_project, update_task, update_milestone, or add_note
- ALWAYS verify before completing a task: call get_task_context(task_id) — every finalized file must have its public functions tracked
- ALWAYS call aifp_end when user says "done" / "wrap up" / "end session"

=== NEVER DO ===

- NEVER read source files before querying the DB — the DB is the index, source files are last resort
- NEVER execute directive workflows from memory — ALWAYS load from DB first
- NEVER mark tasks/milestones complete if code has stubs, placeholders, TODOs, or non-functional work — "complete" means functional and tested, not "file exists." If deferred: create follow-up task, add_note referencing specific unfinished files/functions
- NEVER finalize files without reserving+finalizing their public functions
- NEVER finalize functions without adding interactions for cross-function dependencies
- NEVER finalize functions without linking types_functions for tracked type usage
- NEVER skip flow assignment — every finalized file needs add_file_to_flow or add_file_flows
- NEVER leave a finalized file with 0 tracked functions (unless data-only/config-only)
- NEVER proceed with non-FP projects — STOP and inform user AIFP is FP-only

=== DURING SESSION ===

- aifp_run(is_new_session=false): checkpoint calls for watchdog alerts — return_statements remind you
- aifp_status(): call when YOUR context is stale/compressed, NOT for user "status" requests. Staleness test: if you cannot recall the current milestone name and active task, your context is stale
- get_supportive_context(): reload detailed reference material if context was compressed
- Helpers FIRST (99%) → orchestrators → direct SQL (last resort, reads only)
- user_directives.db is the exception — allows free SQL

=== WHAT IS AIFP ===

Three principles:
1. Functional Procedural — all code must be FP-compliant (NON-NEGOTIABLE)
2. Directive-Driven — workflows loaded from DB tell you WHEN and WHAT
3. Database-Driven — DB is source of truth, not your memory

Four databases: aifp_core.db (read-only directives/helpers), project.db (files/functions/tasks),
user_preferences.db (settings), user_directives.db (Case 2 automation only).

=== LIFECYCLE ===

  init → discovery → [progression: one task at a time] → completion → end

File coding loop (every file):
  reserve file → reserve functions+types → write FP code → finalize file
  → assign to flow(s) → finalize functions+types → add interactions → add types_functions

Tasks created incrementally as work progresses, NOT all at once.

=== BE PROACTIVE ===

- Use project state to drive action — do NOT wait for commands
- Pending tasks → present with priority, execute or await user choice
- New user request → route through project_task_decomposition
- project_continue_on_start=true → auto-continue work
- Supportive context (loaded by aifp_run) has detailed reference for all workflows
