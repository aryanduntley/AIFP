AIFP SUPPORTIVE CONTEXT — CONCISE REFERENCE
=============================================

Supplements the system prompt. Loaded on session start via aifp_run(is_new_session=true).
Reload anytime via get_supportive_context() if context feels stale or compressed.

=== 1. FP RULES — DIRECTIVE REFERENCE ===

66 FP directives in 18 categories cover all FP patterns in detail.
Call get_fp_directive_index() or search_directives(type='fp', keyword='...').

Key categories: purity, immutability, error_handling, composition, side_effects,
anti_oop, type_system, testing, concurrency, optimization, state_elimination,
wrapper_generation, result_types.

Quick rules:
- Pure functions: same inputs → same outputs, no side effects, no I/O inside logic
- Read-only global constants (Final) encouraged; mutable globals → use state database
- Immutability: frozen dataclasses, tuples, frozenset — return new copies, never mutate
- No OOP: frozen dataclass + standalone functions only, no classes with methods
- Wrap external libraries in Result-returning pure wrappers
- Result/Option types for errors, not exceptions for control flow

=== 2. STATE DATABASE USAGE ===

Auto-created at <source-dir>/.state/runtime.db. Replaces mutable globals.

Operations: set_var(key, value), get_var(key), delete_var(key), increment_var(key, amount)
Use for: counters, toggles, runtime config, application state.
Don't use for: sessions, rate limits, job queues (create separate project-specific DB).
Extensible: AI can add tables for edge cases.
See fp_state_elimination.md for detailed patterns.

=== 3. DRY PRINCIPLE ===

Extract common utilities at the HIGHEST appropriate scope level.

Extract when: code IDENTICAL across 2+ files, single responsibility, no conditionals needed.
Don't extract when: similar but not identical, would require many parameters, would create god functions.

Scope levels:
- Global: src/aifp/_common.py (ALL categories)
- Category: src/aifp/helpers/{category}/_common.py
- File: keep in file (one file only)

Decision: 2+ files AND identical → extract immediately. Promote as commonality grows.

=== 4. PROJECT DISCOVERY & PROGRESSION ===

**Project Discovery** (after init, before coding):
Bridge between initialization and active development. Collaborative conversation.

  1. Assess state: Check for pre-existing code, blueprint status
  2. If pre-existing FP code: Delegate to project_catalog first
  3. Blueprint: Discuss purpose, goals, scope, constraints with user
  4. Infrastructure: Confirm detected values, create state database (.state/runtime.db)
  5. Themes/Flows: Identify logical groupings and workflows
  6. Completion Path: Define major stages
  7. Milestones: Break stages into concrete, verifiable milestones (NO tasks yet)

  Key: Ask, don't assume. User knows their project better.
  Routes to aifp_status → project_progression for first task creation.

**Project Progression** (the work loop):
  - ONE task (with items) created at a time per milestone
  - Priority-based work hierarchy: sidequests > subtasks > tasks
  - **BEFORE starting any task**: check get_all_flows() / get_task_flows(task_id) for relevant
    workflow descriptions. Flows describe the intended architecture and user-facing behavior.
    Also check get_all_themes() to orient on which project area the task belongs to.
  - **DURING task work**: use DB lookups FIRST (get_function_by_name, get_type_by_name,
    get_functions_by_file) before reading source files. The DB is the queryable index.
  - Task complete → check if milestone complete → create next task or next milestone
  - Milestone complete → open next milestone → create first task
  - Themes/flows evolve → review completion path
  - Repeat until all stages complete

**Post-Completion Development Cycles**:
  - Two default paths created during discovery: "Added Features" (order 998) and "Updates" (order 999)
  - Both marked 'completed' by default — they don't block initial project completion
  - When post-completion work needed: reopen the appropriate path (status → 'in_progress')
  - Reopening reverts project completion status to incomplete
  - Milestones, tasks, items created on-demand within the reopened path
  - Same progression rules apply — one task at a time, standard workflow
  - When done, path completes again → project returns to complete status

  State machine: aifp_status detects current state → project_progression determines
  what to create next → execute → loop back to status.

  Tasks are created incrementally as work progresses, NOT all at once.
  Each completion triggers the next creation.

=== 5. DIRECTIVE EXECUTION ===

**CRITICAL: ALWAYS load directives before executing them.**
Call get_directive_by_name(name) to get the current workflow steps from the database.
Do NOT execute directive workflows from memory or training data — directives evolve and
the database is the source of truth. Cached knowledge from prior sessions may be stale.

**Execution Order** (for ALL directives):
1. Load directive: get_directive_by_name(directive_name) → read workflow steps
2. User preferences loaded FIRST (user_preferences_sync auto-called if customizable)
3. Preferences applied to directive parameters
4. Directive executes with modified behavior per loaded workflow steps
5. Results returned

**Trunk → Branches → Fallback Pattern**:
- trunk: Main analysis step
- branches: Conditional paths based on analysis results
- fallback: Default action if no branch matches

**Directive MD Documentation** (reference layer):
- Location: src/aifp/reference/directives/{category}/{directive_name}.md
- Contains: Complete workflows, examples, edge cases, FP compliance notes, cross-directive relationships
- When to read: Complex edge cases, ambiguity in workflow, FP compliance questions, cross-directive relationships
- Access: get_directive_content(directive_name) returns full MD documentation
- Directives have guidance for ALL aspects of AIFP — file operations, interactions, types,
  task management, FP compliance, git, user preferences, reserve/finalize, and more.
  If you're uncertain about a workflow, there is almost certainly a directive for it.

**Directive Flows** (navigation system — distinct from project themes/flows):
- Stored in directive_flow table in aifp_core.db
- Define "what directive comes next" based on current state
- Query: get_flows_from_directive(directive_name) — returns directive flow navigation data
- All completion directives loop back to aifp_status
- NOTE: "Directive flows" (navigation between directives) are separate from "project flows"
  (user-facing behavior/architecture stored in flow_themes/file_flows tables in project.db).
  Do not confuse the two.

=== 6. PROJECT FILE WRITE WORKFLOW ===

**FIRST**: Call get_directive_by_name("project_file_write") to load the current workflow steps.
Directives evolve — do NOT execute from memory. The database is the source of truth.

When writing any file:
1. Reserve file/function/type names via project_reserve_finalize → get database IDs
2. Write FP-compliant code with IDs embedded in names
3. Call project_file_write directive → updates project.db (files, functions, interactions tables)
4. Finalize reserved names via project_reserve_finalize
5. Add interactions for ALL finalized functions that call/chain/compose/pipe other tracked functions
6. Add types_functions linkages for functions that consume or produce tracked types

Never write code without tracking it in project.db.
Never finalize functions without adding interactions and types_functions linkages.

=== 7. RESERVE → WRITE → FINALIZE → INTERACT → LINK FLOW ===

Purpose: Get database IDs BEFORE coding so names are instantly queryable.

**FIRST**: Call get_directive_by_name("project_reserve_finalize") to load the current workflow.
Directives are the authoritative source — do NOT execute from memory alone.

1. Reserve: file → file_id, functions → function_ids (PUBLIC only), types → type_ids
2. Embed IDs: filename_id_42.py, function_name_id_99(), TypeName_id_15
3. Write: FP-compliant code with embedded IDs
4. Finalize: mark reserved → finalized, record path/line numbers/dependencies
5. Interactions: add_interaction/add_interactions for EVERY function that calls/chains/borrows/composes/pipes another tracked function
6. Types_functions: add_types_functions for every function that consumes or produces a tracked type

Batch helpers available (reserve_files, finalize_files, add_interactions, add_types_functions).

MANDATORY COUPLING — files, functions, types, interactions, AND types_functions are tracked TOGETHER:
- The complete flow is: reserve file → reserve functions + types → write code → finalize file → finalize functions + types → add interactions → add types_functions
- Never finalize a file without also reserving+finalizing its public functions AND exported types
- Never finalize functions without adding interactions for cross-file dependencies
- Never finalize functions without linking types_functions for tracked type usage
- A finalized file with 0 tracked functions or untracked types is INCOMPLETE (unless data-only/config-only)
- A finalized function with imports from tracked files but 0 interactions = INCOMPLETE
- Return statements from these tools carry mandatory next-step guidance — DO NOT ignore them

INTERACTIONS ENFORCEMENT:
- After finalizing functions, check: does this function call/chain/borrow/compose/pipe any other tracked function?
- If yes → add_interaction(source_name, target_name, type) or add_interactions([(source_id, target_id, type, desc), ...])
- Types: 'call' (direct invocation), 'chain' (sequential composition), 'borrow' (uses return value), 'compose' (function composition), 'pipe' (data pipeline)
- Interactions are how the AI understands the dependency graph. Without them, refactoring impact is blind.
- Do NOT batch-finalize functions and skip interactions — the finalize → interact flow is atomic.

TYPES_FUNCTIONS AUTO-INFERENCE (after function finalization):
- Scan finalized function's parameters and returns for tracked type names
- If matches found, auto-infer role and call add_types_functions:
  - appears only in returns → 'factory'
  - appears in params AND returns (same type) → 'transformer'
  - appears in params, returns boolean/Result → 'validator'
  - appears only in params → 'accessor' or 'operator'
  - Complex cases: 'pattern_matcher' (destructures variants), 'combinator' (combines instances)
- Auto-inference covers ~80% of cases. Correct or add remaining roles manually.
- This linkage answers "what functions use TypeX?" without scanning all files.

FIELD POPULATION IS MANDATORY:
- Functions: purpose (what it does, 5-10 words), parameters (full signature string), returns (return type string)
  Example: purpose="Build UI coin selection entry from CoinEntry", parameters="coin: CoinEntry", returns="CoinSelectionEntry"
- Types: definition_json (full type shape as JSON), description (what it represents, 5-10 words)
  Example: definition_json={"kind": "union", "variants": [...]}, description="Payment recipient — handle or direct address"
- These fields ARE the database's value. Without them, the DB is just a filename index.
- Null fields make entities invisible to future AI sessions. Populate at reserve time, verify at finalize time.

DB-FIRST LOOKUP POLICY:
- Need a function signature? → get_function_by_name FIRST
- Need a type shape? → get_type_by_name FIRST
- Need to know what's in a file? → get_functions_by_file FIRST
- Need file metadata? → get_file_by_name or get_file_by_path FIRST
- Only read the source file if the DB doesn't answer your question
- This saves context window space and is faster than reading full files
- The DB is the index; source files are the full text

EXCEPTIONS — skip ID embedding:
- Files: __init__.py, .db files, pyproject.toml, setup.py, conftest.py
- Private functions (_underscore prefix): do NOT reserve or track

=== 8. USE CASE 2: CUSTOM DIRECTIVE AUTOMATION ===

9 user_directive_* directives for full pipeline.
Call search_directives(keyword='user_directive') for details.

Pipeline: parse → validate → implement → approve → activate → monitor → update → deactivate → status.

Two use cases (never mixed):
- Case 1: Regular software development (NULL user_directives_status)
- Case 2: Automation projects (non-NULL user_directives_status)

Detection during discovery: automation BEHAVIOR signals ("when X do Y", trigger, schedule)
vs software BUILD signals ("a web server", "a library").

Status progression: NULL → pending_discovery → pending_parse → in_progress → active → disabled.
Key: Implementation phase IS Case 1 development — AI builds software to execute user directives.

=== 9. USER PREFERENCES SYSTEM ===

Atomic key-value overrides for directive behavior.
Created when users express preferences ("Always add docstrings", "Use 2-space indentation").
Auto-loaded before customizable directives (user_preferences_sync).

Creating: user_preferences_update → directive_name.preference_key = value.
Tracking features: ALL OFF by default. 5 features for dev/debugging only.
Regular users: assume tracking always disabled. Don't mention tracking to regular users.

=== 10. GIT + FP COLLABORATION ===

FP makes Git superior: pure functions easy to test, no hierarchy conflicts,
immutable data fewer state conflicts, isolated side effects easy identification.

Git directives: git_init, git_detect_external_changes, git_create_branch,
git_detect_conflicts, git_merge_branch. Branch naming: aifp-{user}-{number}.

=== 11. EVOLUTION NOTES ===

Any change to ProjectBlueprint.md, completion paths, themes, flows, or milestones
MUST be accompanied by a notes table entry describing what changed and why.

Use add_note with:
- note_type='evolution' (structural/architectural) or 'task_context' (status changes)
- source='ai', directive_name= triggering directive, severity='info'
- Do NOT set reference_table for blueprint notes (it's a file, not a DB table)

Blueprint notes: note_type='evolution' + directive_name='project_blueprint_update'.
Non-negotiable — the database is the project's memory.

=== 12. EDGE CASES & RECOVERY ===

If confused, lost context, or unsure what to do next:
- aifp_status() → fresh project state from database
- get_supportive_context() → reload this reference material
- search_directives(keyword="...") → find the right directive
- project_notes_log → record decisions, confusion, context
- project_sidequest_create → track unexpected work
- aifp_run(is_new_session=true) → full reload of everything

Trust the database over cached context.

Non-FP projects: If existing OOP codebase detected → STOP, inform user AIFP is FP-only.
Offer: disable AIFP, or start new FP project, convert to FP (major refactor - discouraged).
DO NOT proceed with managing non-FP projects. DO NOT try to convert unless explicitly requested

=== 13. SESSION MANAGEMENT ===

**Startup** (no context):
  aifp_run(is_new_session=true) → full bundle: status, settings, directives, supportive context, guidance
  Cache bundle → check state:
  - .aifp-project/ missing → check .git/.aifp/ backup → offer restore or aifp_init
  - .aifp-project/ exists → read ProjectBlueprint.md → present status + next actions
  - project_continue_on_start=true → auto-continue; otherwise → await user direction
  Directive names cached from bundle. Use get_directive(name) or search_directives(type, keyword).

**Continuation** (every interaction after startup):
  aifp_run(is_new_session=false) → lightweight guidance
  Execute → update DB → loop. Database handles state, not your memory.

**Mid-session refresh** (AI-initiated):
  aifp_status() → fresh project state + supportive context (lighter than full bundle)
  get_supportive_context() alone if you only need this reference reloaded

**Session End** ("done"/"wrap up"/"I'm done"):
  aifp_end → audit tracked files/functions, stop watchdog, log session summary
  If user closes without aifp_end → watchdog catches drift next session

**Use available tools**
**DB policy**: helpers FIRST → orchestrators → directives → direct SQL (last resort, reads only).
Exception: user_directives.db allows free SQL.

=== 14. FULL WORKFLOW EXAMPLE (reference — rely on directives for definitive steps) ===

Example: user says "help me create a calculator"

PHASE: SESSION START
  1. has context? no → aifp_run(is_new_session=true) → full bundle
     has context? yes → aifp_run(is_new_session=false) → lightweight guidance
  2. from bundle: cache directive names, settings, project status, supportive context
  3. need directive details? → get_directive_by_name(directive_name, project_root)
     need deeper context? → read md file (path in directive's md_file_path field)
  4. need directive routing? → get_flows_from_directive(directive_name) → condition-based next step

PHASE: INITIALIZATION (status shows .aifp-project/ missing)
  5. route: aifp_status → aifp_init (directive flow: aifp_initialized=false)
  6. Phase 1 — call aifp_init(project_root) helper → creates dirs, DBs, blueprint template, .git if missing (returns git_status)
  7. Phase 2 — AI detects language/tools, prompts user for name/purpose/goals
     → FIRST: call create_project(name, purpose, goals, status, version) to insert project row
     → THEN (parallelizable): update_source_directory, update infrastructure entries (value field only)
     → THEN: populate ProjectBlueprint.md
  8. route: aifp_init → project_discovery (directive workflow: next_directive="project_discovery")
  9. discovery — execute git_init directive (if git available), then collaborate with user:
     → git_init: gitignore, initial commit, hash storage (based on git_status from Phase 1)
     → blueprint refinement, themes, flows, completion path, milestones (NO tasks yet)
     → populate DB via add_completion_path, add_milestone, etc.
     → add_note(note_type='evolution', directive_name='project_discovery') for decisions
  10. route: project_discovery → aifp_status → project_progression

PHASE: WORK LOOP (project_progression state machine)
  11. identify active completion_path step via get_all_completion_paths
      if none active → update_completion_path(id, status='in_progress') on first open
  12. identify active milestone via get_milestones_by_completion_path
      if none active → update_milestone(id, status='in_progress') on first open
      if none exist → create milestones for this path step
  13. identify active work — priority order: sidequest > subtask > task
      if no active task → create ONE task for milestone:
        add_task(milestone_id, name, flow_ids, priority)
        then create items: add_project_entry(table='items', data={reference_table:'tasks', reference_id, name, description})
  14. work on items:
      a. need new file?
         → reserve_file(name, path, language) → returns {id, id_in_name}
           (use skip_id_naming=true for __init__.py, .db, config files)
         → write file with _id_XX in name (unless skip_id_naming)
         → finalize_file(file_id, name, path, language)
      b. plan code → reserve functions (single or batch):
         → reserve_function(name, file_id) or reserve_functions([...]) → returns IDs
         → only PUBLIC functions — private _underscore functions not tracked
      c. write FP-compliant code with _id_XX in function names
      d. finalize: finalize_function(function_id, name, file_id) or finalize_functions([...])
      e. item done → update_project_entry(table='items', id, data={status:'completed'})
         → next item → repeat 14a-e
  15. all items done → update_task(id, status='completed')
      → add_note(note_type='task_context', content='task completed: ...')
  16. more tasks needed for milestone?
      yes → create next task (goto 13) — tasks created ONE at a time
      no  → update_milestone(id, status='completed')
  17. more milestones for this path step?
      yes → activate next milestone (goto 12)
      no  → update_completion_path(id, status='completed')
  18. more path steps?
      yes → activate next step (goto 11)
      no  → project_completion_check → project_archive → done

  Alternative: update_project_state(project_root, action, target_type, target_id, data)
    actions: 'start_milestone', 'complete_milestone', 'start_task', 'complete_task',
             'start_path', 'complete_path' — single orchestrator for common state changes

PHASE: MID-WORK CHANGES (can occur at any point during work loop)
  19. user changes scope/goals/themes/infrastructure:
      → discuss with user → update ProjectBlueprint.md if needed
      → add_note(note_type='evolution', directive_name='project_blueprint_update')
      → evolve themes/flows if needed → add_note per change
      → evolve completion path if needed (add new paths AFTER current state only)
      → route: project_evolution → project_progression (reviews path adjustments)
  20. unexpected unrelated work discovered:
      → add_sidequest(paused_task_id, flow_ids, name, priority='critical')
      → create items for sidequest via add_project_entry(table='items', data={reference_table:'sidequests', ...})
      → complete sidequest before resuming paused task
  21. work related to current task but shouldn't modify it:
      → add_subtask(name, parent_task_id, priority='high')
      → create items via add_project_entry(table='items', data={reference_table:'subtasks', ...})
      → complete subtask before parent task

PHASE: USER PREFERENCE CHANGES
  22. user changes how a directive behaves (e.g., "always add docstrings"):
      → user_preferences_update directive
      → table: directive_preferences (directive_name, preference_key, preference_value)
      → add_note(note_type='decision')
  23. user changes project-wide AI behavior (e.g., "continue on start"):
      → user_preferences_update directive
      → table: user_settings (setting_key, setting_value)
      → add_note(note_type='decision')
  24. user requests custom helper behavior extension:
      → table: custom_return_statements (helper_name, statement, active)
      → add_note(note_type='decision')

PHASE: USE CASE 2 — CUSTOM DIRECTIVE AUTOMATION (if detected during discovery)
  Detection: user describes automation BEHAVIOR ("when X do Y") not software to BUILD
    → present Case 1/2 choice → set project.user_directives_status='pending_discovery'
  Status: NULL → pending_discovery → pending_parse → in_progress → active → (disabled)
  Directive pipeline (see directives-user-system for full workflow details):
    25. project_discovery (Case 2 context) → conversational onboarding (pending_parse)
    26. user_directive_parse → user_directive_validate → status='in_progress'
    27. user_directive_implement — uses WORK LOOP above (phases 11-18):
        same reserve → write → finalize, tracked in project.db, code goes to src/
    28. user_directive_approve → user_directive_activate → status='active'
    29. user_directive_monitor (ongoing) | user_directive_update (if source files change)
        changes trigger full re-pipeline: deactivate → parse → validate → implement → approve
